0:00–0:05Why Git, mental model

Snapshots, not diffs.
A commit is a photo of all your files at that moment. Not a list of changes.

Distributed: every clone is a full repo.
When you clone, you get the entire history. You can commit, branch, and view log offline. Push and pull are just sync.

Git is not Dropbox. It doesn't sync continuously. You decide when to snapshot, you decide when to share.
Give a brief history of version control tools. Git provides a workflow for distributed development. Clean branching and merging workflow. Every clone is a full snapshot of the content at *some* point in time. We'll go into a bit more depth on each of these during the preso. First! Let's start a git repo! `git init`

0:05–0:15The three states

The working paradigm: - Viewer: working directory — these are just the files on disk you're looking at and editing. - Make changes: staging — show examples of adding directories, adding files to directories, changing content. - Commit: adding a snapshot to the repository. Use `git status` throughout to show how changes move between states.

0:05–0:15Live demo: first commits

mkdir hello-git && cd hello-git
git init
git status                       # "no commits yet"

echo "# Hello Git" > README.md
git status                       # README.md is "untracked"

git add README.md
git status                       # now "staged"

git commit -m "Initial commit"
git status                       # "nothing to commit, working tree clean"

git log                          # one commit, with hash, author, message
Run this live. Predict what `git status` will say BEFORE running it each time — that's the lesson. After this, edit the README, repeat: status, add, status, commit, status, log. The repetition IS the teaching. If you're running long, skip `git log` here — you'll come back to it in the branching segment.

0:15–0:25Two ways to picture a branch

The mental model is what you reason with. The real view is what Git actually stores.

Two views, same data. Both useful for different things. LEFT (mental model): parallel tracks. Feature branches off main, gets some commits, then merges back. This is how to THINK about branches in your head — a separate lane of work that rejoins later. RIGHT (real view): all Git is actually storing. Four commits in a chain (each pointing to its parent), and two labels — main and feature — that each point to a commit. That's it. No "branches" as separate things. The right view kills the "branches are copies" misconception. Branches cost nothing because they're just labels. Note that the right view here shows the state BEFORE a merge — both main and feature are just labels on the same chain. Once you merge, the labels point to the same commit. We'll see that on the next slides.

0:15–0:25Live demo: branching

git switch -c feature-greeting   # create + switch to branch
echo "Hello, world!" >> README.md
git add README.md
git commit -m "Add greeting"

git switch main                  # back to main; greeting gone
git merge feature-greeting       # fast-forward (next slide)

git log --graph --oneline --all  # see the picture

Tip: alias the log command — git config --global alias.lg "log --graph --oneline --all"

Run it live. The first merge will be fast-forward — show the graph BEFORE explaining what just happened. Alias tip: tell them they'll use `git lg` constantly. It's the fastest way to develop a feel for branch shape.

0:15–0:25Fast-forward merge

Main has no new commits — Git just slides the pointer forward

BEFORE you advance the slide: ask the room "what do you think will happen when I merge feature into main?" Then advance. Main slides forward to point at C4. NO new commit was created. That's a fast-forward. Why? Main hadn't moved since the branch was made. Git can just walk the pointer forward.

0:15–0:25Real merge commit

Both branches moved forward — Git creates a new commit with two parents

Same setup as the previous slide, but main also moved forward (C3, C4) while feature was being worked on (C5, C6). Git can't just walk a pointer — both branches have new work. So it creates a NEW commit M (amber) with TWO parents. That's the diamond shape. Press O for overview mode if you want to flip between the FF slide and this one to drive the contrast home.

0:25–0:35git pull = git fetch + git merge

`origin/main` is your LOCAL CACHE of where the remote's main was last time you checked. `git fetch` updates the cache (origin/main) — it does NOT change your `main`. `git merge origin/main` then combines that into your local main. `git pull` does both. Most of the time you want pull.

0:25–0:35The most common workflow hiccup

You try to push. Git rejects it because the remote has commits you don't have.

git push
# ! [rejected]        main -> main (fetch first)
# error: failed to push some refs to '...'

git pull       # get + merge their work
git push       # now succeeds
The rule: if push is rejected, git pull first, then push again.
This will happen to them on day one. Demystify it now. To demo: edit a file via the GitHub web UI in another tab, then try to push from your terminal. Push fails. Pull. Push again.

0:35–0:45The PR workflow loop

What their week actually looks like. Walk through it once, then run it live. `git push -u origin ` — only the first push of a new branch. The `-u` sets the upstream so future pushes are just `git push`.

0:45–0:55Undo recipes

Git almost never actually loses your work.

Five recipes. Then a safety net.

The most valuable segment in the hour. Reframe: "Git almost never loses your work" — this is true and changes their whole posture toward the tool. Exceptions are `--hard` and force-push. Both flagged when we get there.

0:45–0:55Which undo do I need?

The most-referenced slide. They'll mentally come back to this for months. Three rows = three scenarios based on the state of your work. `--hard` (red) is the ONLY undo that actually deletes uncommitted work. Say it out loud. Twice. Walk each row with a mini-demo. Then `git reflog` last — show recent HEAD movements, recover a "lost" commit.
